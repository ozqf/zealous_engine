## Architectyre Notes

### Merge Client/Server modules

* Original plan of making this a networking test is kind of in the bin!
    * Merge separated client/server into a single 'game' module with 
        client/server as specific logic to manipulate the game instance.
* Primary goals
    * Game can be recorded and played back.
    * Server/Client logic is kept out of the game update itself.

Game module responsibility is to tick the simulation. As input it takes commands.
Input is generated by the 'client'. Game events like spawn/delete entites are generated
during the game loop and run the next frame.

Generated events are written to a post frame buffer.
* If the game is a client (replaying) all events except input are ignored and read from packets instead.
* If the game is a server output is replicated to clients as well as being processed next frame.

### Asset DB

Issues/Critical gaps:
* Asset db does not support generic byte data or text, only specific supported types.
* Does not store file names safely but just as pointers!
* Asset db access is currently unsafe between renderer and app.
* Dirty assets are detected and updated at render start.
    * Entire asset db is scanned even if only one asset is dirty - store a list of dirty assets, if    
        list overflows then flag entire db for a scan.
    * App could be updating and flagging geometry as dirty during render pass? Shouldn't
    matter as asset upload is performed at one time only (currently).
* Basic storage/retrieval functionality is duplicated between each asset type:
    should be made more generic
* Managing of asset <-> GPU handles should be done within the renderer, not the asset db.
    Currently handles have to be stored in the asset header.
